## レビュー観点

### 1. ケース一覧の網羅性

- Happy / Edge / Error の 3 区分すべて埋まっているか
- 想定される境界値・並行・例外パスが過不足なく拾われているか
  - 入力: 空 / 極小 / 極大 / 不正型 / 並行
  - 状態: 未初期化 / 初期化済 / 終了済
  - 出力: 正常 / エラー / 部分成功
- 1 行 1 ケースの粒度を維持できているか（複合ケースは分割されているか）
- describe / it タイトルが意図を読み取れる粒度か

### 2. 責務分担マトリクスの妥当性

- Unit / E2E の判断根拠が明確か
- 純粋ロジックを E2E に振っていないか（過剰結合 / コスト超過）
- DOM 描画・実ブラウザ依存を Unit に振っていないか（検証不能）
- 重複検証（Unit と E2E の両方で同じ振る舞い検証）になっていないか

### 3. テスト規約との整合

ワークフローの `testing` policy（Unit/E2E 判定基準・Happy/Edge/Error 区分・記述スタイル・ヘルパー活用ガイド）に整合しているか。具体的には:

- プロジェクトの既存テストヘルパーで書ける前提が明示されているか
- 命名（describe / it タイトル、ファイル名規約）が `testing` policy の規定に揃うか
- TDD サイクル（Red→Green→Refactor）の Red 段階で書ける粒度になっているか

### 4. 不確定要素の取り扱い

- 「要ユーザー確認」事項が末尾に列挙されているか
- 設計表に推測値が混入していないか

## アウトプット (test-design-review.md)

- 冒頭で `approved` または `needs_fix` を明記

  - 対象ケース ID または セクション番号
  - 問題（観点 1〜4 のどれに該当するか明記）
  - 修正提案（具体的に）
