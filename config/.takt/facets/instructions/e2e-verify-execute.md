# E2E 動作確認 — 実行

計画ステップで作成したシナリオに従い、Chrome DevTools MCP でブラウザを操作して動作確認を実行する。

## 前提

- Chrome が起動済みであること（autoConnect モード）
- 計画レポート（e2e-verify-plan.md）のシナリオを順に実行する

## 手順

### 1. dev server 起動

```
計画で特定したコマンドを Bash で実行（バックグラウンド起動）
→ ヘルスチェック: curl -s -o /dev/null -w "%{http_code}" http://localhost:<port>
→ 200 が返るまで最大 30 秒待機（5 秒間隔でリトライ）
```

起動に失敗した場合はエラー内容を記録して ABORT。

### 2. ブラウザ操作の基本フロー

各シナリオについて以下を繰り返す:

1. **ナビゲーション**: `new_page` または `navigate_page` で対象 URL を開く
2. **読み込み待機**: `wait_for` でページの主要コンテンツが表示されるまで待つ
3. **初期状態のスクリーンショット**: `take_screenshot` で操作前の状態を記録
4. **ページ構造の取得**: `take_snapshot` で要素の `uid` を取得
5. **操作実行**: `click`, `fill`, `select_option` 等で計画の操作を実行
6. **結果確認**: 操作後に `take_snapshot` + `take_screenshot` で状態を記録
7. **判定**: 期待結果と実際の状態を比較し、PASS / FAIL を記録

### 3. Happy パスの検証ポイント

- 画面遷移が正しいか（URL の変化）
- 表示内容が期待通りか（テキスト、要素の存在）
- フォーム送信後のフィードバック（成功メッセージ、リダイレクト）
- データの永続化（リロード後も表示されるか）

### 4. エラーケースの検証ポイント

- バリデーションエラーメッセージが表示されるか
- エラー状態の UI フィードバック（赤枠、アイコン等）
- エラー後にフォームの入力値が保持されるか
- 404 ページが適切に表示されるか

### 5. スクリーンショットの命名規則

`filePath` パラメータで保存先を指定する:

```
.takt/runs/<run_slug>/screenshots/<scenario-id>-<step>.png

例:
- H-1-01-initial.png（Happy パス 1 の初期状態）
- H-1-02-after-input.png（入力後）
- H-1-03-result.png（送信結果）
- E-1-01-empty-submit.png（空送信）
- E-1-02-validation-error.png（バリデーションエラー表示）
```

### 6. 失敗時の対応

- 1 シナリオが FAIL しても残りのシナリオは続行する
- FAIL したシナリオはスクリーンショットと実際の状態を詳細に記録する
- dev server のクラッシュやブラウザ接続断は即座に ABORT

### 7. クリーンアップ

全シナリオ完了後:
- 開いたタブを `close_page` で閉じる（自分が開いたもののみ）
- dev server プロセスを停止する
