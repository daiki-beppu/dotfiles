# article-design

GitHub Issue の粗い記事アイデアを、執筆に入れるレベルの詳細な記事設計まで煮詰める。

## When to Use

- Issue にある 5 行程度の構成案を詳細化したいとき
- `/article-design <Issue番号>` コマンドを実行したとき

## Task

### Step 1: コンテキスト収集

以下を並行して実行する：

```bash
gh issue view <number> --json title,body,labels
```

- `references/design-example.md` を読み込む（手本となる詳細設計）
- `knowhow-library/` 配下に既存記事があれば 1 本読み、トーン・粒度を把握

引数で Issue 番号が指定されなかった場合は、ユーザーに Issue 番号を尋ねる。

### Step 2: 分析・深掘り

Issue の構成案を読み取り、以下の観点で深掘りする：

1. **各セクションの具体化ポイント**を洗い出す
   - 「実践」「応用」が抽象的なら、具体的な操作手順・シナリオに落とす
   - 会話例やコード例をどこに入れるか、方針を決める

2. **対象機能に関連する素材**をコードベースから探す
   - CLAUDE.md、スキル定義、設定ファイルなど実際のコードや設定
   - 記事で参照できそうなファイルパスをリストアップ

3. **ユーザーへの質問**（必要に応じて）
   - 実体験・具体的エピソードの有無
   - 想定読者のレベル感
   - 掴みの方向性

### Step 3: 詳細設計の生成

`references/design-example.md` と同じ粒度で記事設計を生成する。
以下の要素をすべて含めること：

- **記事メタ**
  - カテゴリ、公開範囲、タグ

- **対象機能**

- **全体構成**（セクションごとに）
  - セクションタイトル（番号付き）
  - 各セクションの具体的な内容を箇条書きで列挙
  - 会話例・コード例の方針（「執筆時に生成する」「実際のコードを引用」等）
  - 太字で強調すべきポイント

- **参照素材テーブル**
  - 記事で引用・参照するファイルのパスと用途

### Step 4: ユーザー確認

生成した詳細設計をユーザーに提示してフィードバックを受ける。
修正→再提示を確定まで繰り返す。

### Step 5: Issue 更新

確定した詳細設計で Issue 本文を更新する：

```bash
gh issue edit <number> --body "..."
```

更新後、Issue の URL を表示する。

## Rules

- 設計は日本語で書く（技術用語・コード・識別子はそのまま）
- 手本（design-example.md）の粒度を下回らない — 各セクション最低 3 つのサブポイント
- 「会話例は執筆時に生成する」等、執筆フェーズへの申し送りを明記する
- ユーザーの確認なしに Issue を更新しない
