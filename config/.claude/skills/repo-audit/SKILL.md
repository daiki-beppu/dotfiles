# repo-audit — リポジトリ全体の AI 残骸スキャン

## Overview

AI にタスクを継続的に任せていると、個別の PR レビューでは見落としがちな「残骸」がリポジトリに溜まる。repo-audit はリポジトリ全体を走査して 4 カテゴリの残骸を検出し、ユーザーに選別させた上で `issue` スキル経由で 1 件ずつ GitHub issue として記録する。

修正は行わない。検出と起票のみ。

## When to Use

- AI が書いたコードや設定が積み重なったタイミングでの定期点検
- 「AI の残骸チェックして」「ドキュメントと実装ずれてへん？」「お掃除したい」など自然文での依頼
- `/repo-audit` コマンドを実行したとき
- `self-review` スキルが差分ベースで扱えない、リポジトリ全体に散らばった問題を洗い出したいとき
- `deprecated` コメントや旧名エイリアスが溜まっていないかを確認したいとき

## 検出カテゴリ

### カテゴリ 1: 過剰な防御 / 抽象化 / 半端な実装

以下のパターンを Grep と Read で確認する。

| 種別 | 検出の目印 |
|------|-----------|
| デバッグ出力残り | `console.log`, `print(`（本番コードで意図不明なもの）, `dbg!`, `debugger;` |
| 到達しないコメント | `TODO`, `FIXME`, `XXX`, `HACK` コメント（ただし `TODO(human)` 等の意図的マーカーは除外） |
| 空の防御 | `catch {}`, `except Exception: pass`, エラーを握りつぶすだけの try/except |
| 未完成実装 | `throw new Error("not implemented")`, `pass  # TODO`, `return null  // later` |
| 無意味な抽象化 | 1 箇所でしか呼ばれていない、元関数を素通しするだけのラッパー関数 |

誤検出防止のため、ヒット箇所は必ず前後行を Read で確認し、意図的な実装か判断する。Grep ヒットだけで issue 化しない。

### カテゴリ 2: ドキュメントの陳腐化

README / CLAUDE.md / 各 SKILL.md の記述が実態と合っているかを突き合わせる。

突合対象ごとに以下を確認する。

| ドキュメント | 突合対象 | 典型的な陳腐化パターン |
|-------------|---------|---------------------|
| `README.md` | `scripts/`, `Makefile`, `package.json`, `flake.nix` | 「`nr build` でビルド」と書いてあるがスクリプト側に build が無い / セットアップ手順が旧バージョンのまま |
| `CLAUDE.md`（プロジェクト直下） | 実ディレクトリ構成、運用ルール、口調指定 | 記述された「config/ 配下に配置」が実際と違う / 削除済みコマンドへの言及が残る |
| 各 `SKILL.md` | 自スキルの description と本文、他スキルへの参照 | description で宣伝してる機能が本文に無い / 例示コマンドの引数が古い |
| `config/.claude/CLAUDE.md` | 他の CLAUDE.md・各スキルとの整合 | グローバル規約と個別スキルのルールが矛盾 |

**重大度の目安**:
- **high**: 参照先（コマンド・ファイル・スキル名）が完全に存在しない
- **medium**: 存在はするが引数や使い方の記述が旧式
- **low**: 表現の古さ・誤字・口調の不一致

**行単位に落とせない "方針ズレ" の扱い**: README 全体で想定する前提（例: OS, 依存ツール）が現状と食い違う場合は、該当見出し単位で 1 件として起票する。行番号は見出しの位置を使う。

### カテゴリ 3: 実装とスキルの乖離

`config/.claude/skills/` 配下の各 SKILL.md が参照する以下が実在するかを検証する。

- 本文中で使う **外部コマンド**（例: `gh`, `nr`, `nlx`, `darwin-rebuild`）が PATH 上に存在するか（`command -v <cmd>` で確認）
- 本文中で言及される **ファイルパス**（例: `config/.claude/settings.json`）がリポジトリに存在するか
- 他スキルを呼ぶ記述（例: `/skills simplify`, `Skill ツールで xxx を呼び出す`）のスキル名が実在するか
- `config/.claude/settings.json` の hooks / permissions で参照されるコマンドが実行可能か

`gh`, `git`, `command -v` は Bash で逐次確認できる。スキル名の実在確認は `ls config/.claude/skills/` と照合する。

### カテゴリ 4: 意味のない後方互換維持

削除済み機能への移行告知、旧名エイリアス、未使用の `deprecated` 分岐など、もう維持する必要がなくなっている互換の残骸を検出する。個別 PR レビューでは拾いにくく、実装側・ドキュメント側の両方に現れるため独立カテゴリとして扱う。

| 種別 | 検出の目印 |
|------|-----------|
| 明示マーカー | TODO(human) — 拾いたいキーワード列をここに列挙する |
| 参照切れの移行告知 | ドキュメント内の「以前は X、現在は Y」という記述で、X が他のどこからも参照されていない |
| 未使用エイリアス / ラッパー | 関数・コマンド・`zsh abbr`・skill 名の旧名定義で、他ファイルからの呼び出しが 0 件 |
| 廃止フラグの残存 | `settings.json` / `flake.nix` / `nix/packages.nix` などで、現行ツールが受け付けないオプションや、削除済み hook / permission への参照 |
| 到達不能な互換分岐 | 「バージョン X 未満なら〜」等の条件分岐で、条件を満たす実行環境がもう存在しない |

**確認手順**:

1. 上記キーワードを Grep で抽出
2. Read で前後を読み、意図（現役の互換維持 vs 残骸）を判定
3. エイリアス・関数・skill 名については `git grep -n <旧名>` で全リポジトリ参照を集計。定義箇所以外でヒット 0 件なら「未使用」と判断
4. `git log -S <旧名>` で、一度削除された参照が復活していない履歴であることを確認
5. 現行ツールの受理オプションと設定ファイルの突合（例: `darwin-rebuild --help`, `claude --help` との照合）

**重大度の目安**:
- **high**: 参照先が完全に存在しない / 分岐が到達不能
- **medium**: 参照はあるが実質呼び出し 0（残しておく価値が失われている）
- **low**: コメントやドキュメント記述だけ残っている

## 実行フロー

### Phase 1: 対象の列挙

```bash
git ls-files                        # 追跡ファイル一覧（.gitignore 尊重）
gh repo view --json nameWithOwner   # リポジトリ名
gh label list --json name           # 既存ラベル（issue 起票時のため前取り）
```

### Phase 2: カテゴリごとに走査

Grep（パターンマッチ）→ Read（前後文脈の確認）→ 判断、の順で進める。各カテゴリの「検出の目印」を出発点にし、必要に応じて `command -v` や `ls` で実体確認まで行う。

### Phase 3: 検出結果の一覧提示

以下のテーブル形式で表示する。

```
## 検出結果 (合計 N 件)

| # | カテゴリ | 場所 | 問題 | 根拠 |
|---|---------|------|------|------|
| 1 | 過剰防御 | src/foo.ts:42 | 空の catch | 例外を握りつぶしている |
| 2 | ドキュ陳腐化 | README.md | `scripts/build.sh` が存在しない | 参照先ファイルが無い |
| 3 | スキル乖離 | config/.claude/skills/pr/SKILL.md | `nr ci` に対応するスクリプトが無い | package.json と突合 |
| 4 | 後方互換残骸 | config/.zshrc:120 | 旧 alias `old-cmd` が誰からも呼ばれていない | git grep で参照ゼロ・コミット履歴上も未使用 |
```

件数が多い場合はカテゴリと重大度でソートする。

### Phase 4: 起票対象の選別

`AskUserQuestion` を multiSelect で使い、ユーザーに起票したい項目を選ばせる。選択肢は「全部」「カテゴリごと」「個別」のいずれにも対応できるよう、件数が 10 件以内なら個別選択、それ以上ならカテゴリ単位の選択を提示する。

### Phase 5: 1 件ずつ起票

選ばれた各項目について、`Skill` ツールで `issue` スキルを呼び出し、以下の情報を渡す。

- カテゴリ名（タイトル冒頭の prefix に使う想定: 例 `[repo-audit] 空の catch が残っている`）
- 場所（ファイルパス:行番号）
- 問題の概要と根拠
- 関連するコード抜粋（数行）

issue 本文の最終的な組み立てと `gh issue create` の実行は `issue` スキルが担当する。repo-audit は種となる情報を渡すだけに留める。

全起票完了後、作成された issue の URL 一覧をまとめて表示する。

## 非対象

- **コードの自動修正**: 本スキルは検出と起票のみ。修正は別セッションで人間または別スキルが担当
- **差分ベースのレビュー**: `self-review` スキルの責務
- **セキュリティ脆弱性の深掘り**: built-in の `/security-review`（手動実行）の責務
- **PR 作成 / マージ / ブランチ操作**

## Rules

- ヒットした全件を無差別に issue 化してはならない。必ず Phase 4 で選別を挟む
- issue 本文の組み立ては自前で行わず、`issue` スキルに委譲する
- Grep ヒットを前後確認せず断定しない（誤検出は信頼を損なう）
- カテゴリ 4 は Grep ヒットだけで判断しない。`git grep -n` / `git log -S` で呼び出しと履歴を確認してから issue 化する
- 現在も意図的に維持している互換（移行期間中のエイリアス、他リポジトリ向けマイグレーションガイド等）は除外する
- 実装修正は行わない。検出報告のみ
- `TODO(human)` など意図的なマーカーはカテゴリ 1 の検出対象から除外する
- **自スキル（`config/.claude/skills/repo-audit/SKILL.md`）自身のルール表記や例示（カテゴリ 1 のパターン文字列 `console.log` `TODO` `catch {}` や、カテゴリ 4 の「明示マーカー」行に列挙された互換キーワード等）は、全カテゴリの検出対象から除外する**
- 「起票はせず一覧だけ出して」と指示された場合は Phase 3 で停止する
